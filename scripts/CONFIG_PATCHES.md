# Правки конфигов ПЕРЕД прогоном (согласование памяти)

Цель: чтобы здоровый стенд НЕ выдавал ложных рекомендаций по конфигу, а Эксп.3
воспроизводился предсказуемо. Источник истины по памяти — лимит контейнера `memory: 6G`.
Значит сервер PostgreSQL рассматривается как машина с 6144 МБ RAM.

## Правка 1 — docker-compose.yml, сервис pg-advisor

```diff
   pg-advisor:
     environment:
       DB_HOST: postgres
       ...
-      TOTAL_RAM_MB: 1024
+      TOTAL_RAM_MB: 6144
       ANALYSIS_INTERVAL_SEC: 60
```
Почему: анализатор считает целевые значения от TOTAL_RAM_MB. При 1024 целевой
shared_buffers = 256 МБ, и сервис либо молчит не по делу, либо считает кэш огромным.
6144 = лимит контейнера → расчёты совпадут с реальностью.

## Правка 2 — configs/postgres/postgresql.conf (НОРМАЛЬНОЕ состояние)

```diff
-shared_buffers = 1GB
+shared_buffers = 1536MB        # 25% от 6144 — здоровое значение, сервис молчит

-effective_cache_size = 3GB
+effective_cache_size = 4608MB  # 75% от 6144 — иначе сервис сочтёт кэш заниженным
```
Почему: при TOTAL_RAM_MB=6144 пороги сервиса таковы, что 1GB shared_buffers уже
на грани срабатывания (порог 1536×0.7=1075 МБ, а 1024<1075 → ложная рекомендация).
1536MB убирает ложное срабатывание; 4608MB согласует effective_cache_size.

work_mem = 32MB — оставь как есть, сервис ругается только если <4 МБ.

## Для Эксперимента 3 (временные значения, потом вернуть)

Фаза «до» (занижено):
```diff
-shared_buffers = 1536MB
+shared_buffers = 128MB
```
Фаза «после» (вернуть норму):
```diff
-shared_buffers = 128MB
+shared_buffers = 1536MB
```
128MB (а не 32MB): при 32MB Postgres может не стартовать стабильно с твоим
max_connections=150; 128MB гарантированно ниже порога рекомендации (1075 МБ) и
заметно роняет cache hit под нагрузкой, но сервер стартует штатно.

## Правка 3 — scripts/00_env.sh

Убедись, что DB_USER совпадает с POSTGRES_USER из .env. Сейчас стоит:
```
export DB_USER="${POSTGRES_USER:-admin}"
```
Если в .env POSTGRES_USER=admin — ок. Если другое — либо экспортируй переменную
перед запуском, либо впиши значение прямо здесь.

## Сверка перед стартом (одна команда)

После `docker compose up -d` проверь, что сервер видит ровно те значения:
```
docker exec -it postgres psql -U admin -d dvdrental -c "SHOW shared_buffers; SHOW effective_cache_size;"
```
Должно показать 1536MB и 4608MB (в нормальном состоянии).

И что анализатор получил правильную RAM (в логах при старте или через рекомендации):
```
docker logs pg-advisor 2>&1 | tail -20
```

## Про запуск на Windows 11 + 16 ГБ

- Гонять в WSL2 (Docker Desktop → Settings → Resources → WSL Integration включить).
- Проект и scripts/ держать в домашней папке WSL (~/), НЕ в /mnt/c/ (медленный I/O).
- Контейнеру отдаём 6 ГБ — на хосте останется ~10 ГБ, закрой тяжёлые приложения.
- Проверь, что WSL2 имеет доступ к ≥7-8 ГБ: при необходимости создай %UserProfile%\.wslconfig:
      [wsl2]
      memory=12GB
  и `wsl --shutdown`, затем заново.
- Jetson Orin Nano НЕ использовать: ARM64 + всего 8 ГБ на систему не дадут честные
  6 ГБ контейнеру, и весь Эксп.3 (легенда «6 ГБ RAM») потеряет смысл.
